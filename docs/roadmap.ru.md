# Roadmap

> Статус: живой документ. Описывает отношение между этим публичным репозиторием и production-кодобазой, а также что планируется к перемещению между ними.

Production-версия `platformctl` живёт в приватном платформенном репозитории. Этот публичный репозиторий — **design-showcase**: он доносит архитектуру и содержит курируемое подмножество кода. Оба сознательно синхронизированы на уровне дизайна, даже когда имплементация расходится.

Этот документ существует, чтобы любой читающий публичный репозиторий — hiring manager, любопытный инженер, потенциальный contributor — знал, что есть сегодня, что планируется и что останется приватным.

---

## Сейчас в публичном репозитории (v0.x альфа)

- Модель спецификации приложения (`internal/appspec`) — общая структура данных, на которой строится каждый onboarding-flow.
- Flow `new-app` / `new-service` / `delete-app` — простейшие write-команды, с dry-run по умолчанию.
- Skeleton `validate` / `render` / `doctor` — достаточно, чтобы продемонстрировать дисциплину read-plane.
- `infra` (skeleton) — placeholder для kubelet-provider операций.
- `export-public` — команда, используемая для зеркалирования частей внутреннего репозитория в этот публичный.
- Примеры под `examples/golden-path/`.
- CI workflows под `.github/workflows/`.
- Apache-2.0 лицензия, `SECURITY.md`.
- Design-документация под `docs/`: этот файл, `architecture.ru.md`, `assist-design.ru.md`, `demo.md`.

## Планируется к публикации (анонимизированно, без внутренних эндпоинтов)

Следующие пункты существуют в приватной кодобазе и будут портироваться в этот репозиторий по мере того, как их можно будет чисто отвязать от внутренней инфраструктуры. Порядок ниже отражает грубый приоритет по портфолио-ценности, а не commitment по срокам поставки.

### Tier 1 — высокий design-сигнал

- **Пакет `outputformat`** — реализация обработки флагов `text` / `json` / `minimal-json`, с примером wiring на паре команд. Самая дешёвая часть для публикации, демонстрирует подход к token-economy.
- **Skeleton `assist` с lexical-retriever** — chunker корпуса, frontmatter-extractor, in-process BM25 retriever, JSON Schema для outputs. Без Qdrant-интеграции, без upstream-corpus, с примером-корпусом из небольшого числа runbook.
- **Harness `assist eval`** — формат golden-question dataset, evaluator, CLI-флаги для порогов. С маленьким примером-корпусом и десятком golden-вопросов; потребители смогут подложить свои.
- **JSON Schemas** — `appspec`, `service-app`, `runbook-frontmatter`, `assist-*` outputs. Схемы переиспользуемы as-is в других проектах.

### Tier 2 — архитектурная полнота

- **Hybrid retrieval interface** — контракт между `assist` и внешним Retrieval API (request/response JSON Schemas, Go-интерфейс). Имплементации остаются pluggable.
- **`docs suggest`** — анализ влияния изменений на документацию. Полезно как CI-шаг в любом docs-heavy репозитории.
- **Подкоманда `config`** — multi-context модель конфигурации.
- **`deploy validate` / `deploy ci-generate`** — валидатор deploy-спецификации и генератор child-pipeline, обобщённые так, чтобы не зависеть от конкретного GitLab-инстанса.

### Tier 3 — существенная cross-cutting работа

- **`hubble` / `logs` / `metrics` / `observe`** — оборачивают внутренние Cilium / Loki / Prometheus эндпоинты и потребуют как развязки, так и примеров-конфигов, чтобы быть полезными публично.
- **`upgrade plan`** — сейчас завязан на специфичный формат Renovate dashboard; обобщённая версия стоит того, чтобы делать, но требует больше абстракции.
- **`new-product`** — самый сложный onboarding-flow; трогает несколько репозиториев и потребовал бы публичного reference layout, чтобы быть осмысленным.

## Останется приватным

Эти куски тесно связаны со специфичной внутренней инфраструктурой и потребовали бы либо (а) такой степени абстракции, что публичная версия перестаёт быть полезной, либо (б) утечки чувствительных операционных деталей.

- **`secrets sync-gitlab-ci`** — соглашения по Vault paths, GitLab project IDs.
- **`registry-ca sync`** — соглашения по именам CA-секретов и layout trust-store, специфичные для внутреннего Harbor.
- **`harbor-robot create`** — соглашения по именам robot-аккаунтов.
- **Lifecycle `runners`** — токены регистрации раннеров, соглашения по именам.
- **`bootstrap`** — bootstrap кластера; целиком environment-specific.
- **Внутренний корпус runbook** — ссылается на внутренние продукты, хостнеймы и инфраструктуру. Схема публична; содержимое — нет.

Дизайн каждого из этих пунктов задокументирован (или будет) в `architecture.ru.md` или в отдельных design-документах.

## Как читать этот репозиторий для hiring-разговора

1. Начни с [README](../README.ru.md) для полного дизайн-контекста и карты команд. Маркеры `[open]` / `[private]` — это правда: не предполагай, что команда, помеченная `[private]`, присутствует в этом дереве.
2. Прочитай [`architecture.ru.md`](architecture.ru.md) для дисциплины read/write plane и паттерна «фабрика манифестов». Это core-идея.
3. Прочитай [`assist-design.ru.md`](assist-design.ru.md) для RAG-подсистемы — corpus contracts, lexical-as-default, CI-gated evaluation harness. Это самая отличительная подсистема.
4. Скачай существующий код `cmd/platformctl` и `internal/appspec`, чтобы посмотреть, как публичный skeleton выкладывает read/write plane.
5. Сверь со своим опытом оператора: какие части этого дизайна узнаваемы, в каких бы возразил, какие бы расширил.

Замысел в том, чтобы интервьюер мог провести двадцать минут здесь и уйти с ясной, фальсифицируемой картиной того, как работает платформенная команда и какова инженерная разумность, — независимо от того, какой production-код стоит за labels.
