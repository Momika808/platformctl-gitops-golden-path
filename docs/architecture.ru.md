# Архитектура

> Статус: design-документ. Описывает полный production CLI; не все подсистемы ниже присутствуют в публичном репозитории.

## Принципы

1. **GitOps-граница.** `platformctl` никогда не мутирует кластер напрямую. Все изменения состояния идут через Git → Merge Request → CI → Flux. CLI — это фабрика манифестов, а не обёртка над kubectl.
2. **Разделение read-plane и write-plane.** Read-команды идемпотентны и без побочных эффектов. Write-команды по умолчанию работают в dry-run; мутация требует явных `--apply --confirm` (или `--auto` в скриптовых контекстах, где оператор уже принял решение).
3. **Schema-first.** Каждый артефакт, который CLI производит или потребляет, описан JSON Schema. Валидация происходит в трёх местах: в самой CLI, в CI и в принимающем репозитории.
4. **Agent-friendly output.** Каждая read-команда поддерживает три формата вывода (`text`, `json`, `minimal-json`), чтобы люди, скрипты и LLM-агенты могли эффективно потреблять одни и те же данные.
5. **RAG как подсистема, а не отдельный сервис.** Retrieval-слой над runbook реализован как набор подкоманд той же CLI, а не как отдельный микросервис. Операторы и AI-агенты пользуются одним и тем же интерфейсом.

## Read-plane

Read-plane существует, чтобы **сделать кластер прозрачным** для людей, скриптов и агентов — без выдачи прав на мутацию. Каждая read-команда:

- идемпотентна
- имеет ограниченный latency (никаких долгих кластерных операций)
- поддерживает три формата вывода
- никогда ничего не мутирует, включая локальную файловую систему, за исключением cache-директорий в `.platformctl-state/`

Самая значимая read-подкоманда — `assist`, подробно описанная в [`assist-design.ru.md`](assist-design.ru.md). Она даёт retrieval-слой над корпусом runbook со стабильным lexical-default и опциональным экспериментальным hybrid-путём для поиска исходного материала.

---

## Write-plane

Write-plane — это место, где важна дисциплина. Каждая write-команда:

- **По умолчанию работает в dry-run.** Без `--apply --confirm` (или `--auto`) она печатает предлагаемый diff и завершается.
- **Никогда не мутирует кластер.** Записи предлагаются как изменения в Git через Merge Request.
- **Сначала валидирует.** JSON Schema валидация запускается до того, как что-либо уйдёт в GitLab.
- **Аудируема.** Каждая write-команда имеет стабильный идентификатор и пишет структурированные логи для корреляции в observability.

Этот дизайн намеренный: стоимость ошибки в write-команде восстановима (закрыть MR, поправить спецификацию, открыть заново) именно потому, что мутация никогда не доходит до кластера напрямую.

## Паттерн «фабрика манифестов»

Онбординг нового сервиса — канонический write-flow:

1. Оператор запускает `platformctl new-app <name>`.
2. CLI загружает шаблоны (embedded), композирует service `appspec`, валидирует против `appspec.schema.json`.
3. Рендерит получившиеся манифесты (Deployment, Service, ServiceMonitor, NetworkPolicy и т.д.) в стандартный layout платформы под `examples/golden-path/`.
4. Считает diff против текущего состояния репозитория `cluster/k8s`.
5. По умолчанию печатает diff. С `--apply --confirm` открывает MR через GitLab API.
6. CI на этом MR заново запускает `validate` + `doctor` + точечные тесты. Ревьюеры мержат.
7. Flux в кластере приводит состояние к желаемому.

CLI таким образом — это **фабрика хорошо сформированных Git-предложений**, а не actor на кластере.

## Agent-friendly output

Каждая read-команда принимает `--output text | json | minimal-json`:

- `text` — default; человекочитаемый, ~50–1400 токенов на типовую команду.
- `json` — полный JSON с API envelope и метаданными; в среднем ~1.91× размер text.
- `minimal-json` — срезает envelope и не-обязательные поля; ~30–50% от полного JSON.

Режим `minimal-json` был добавлен после того, как было измерено, что агентские workflow (Claude, Codex) рутинно вызывают подкоманды CLI десятки раз за задачу. Срезание per-call payload на 60–78% даёт измеримую экономию токенов без потери структуры, которой агенты реально пользуются.

## RAG-подсистема (краткое описание)

См. [`assist-design.ru.md`](assist-design.ru.md) для полного дизайна. Кратко:

- Корпус — это директория runbook платформы, каждый runbook размечен структурированной YAML-frontmatter (`id, title, domain, component, lifecycle, status, class, safety, retrieval_priority`).
- Default retrieval-путь — **lexical BM25**, намеренно выбранный за стабильность и аудируемость в bootstrap-контекстах.
- Существует **экспериментальный hybrid-путь** для поиска исходного материала по большим корпусам; результаты трактуются как evidence pointers, не как инструкции.
- В CI работает **harness eval по golden-вопросам**: 37 курированных запросов с ожидаемыми целевыми путями/секциями; флаги `--min-recall-at-3` и `--max-p95-latency-ms` валят пайплайн при регрессии.

## Что намеренно не в этом репозитории

- Вендор-специфичные клиенты (вызовы Qdrant SDK, построение запросов Loki/Prometheus) описаны, но не выложены, потому что они светят внутренние эндпоинты и потребовали бы плейсхолдеров конфига, которые затемняют дизайн.
- CI-шаблоны, которые связывают CLI с конкретными внутренними инстансами GitLab и Harbor, описаны в `roadmap.ru.md`, но не закоммичены здесь.
- Внутренние хостнеймы, IP-диапазоны и имена продуктов намеренно убраны из этого репозитория.

Цель этого репозитория — донести **архитектуру**, а не быть turnkey-инсталляцией.
