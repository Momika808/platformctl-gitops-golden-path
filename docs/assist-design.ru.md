# `assist` — дизайн RAG-подсистемы

> Статус: design-документ. Реализация работает в приватной production-кодобазе.

`platformctl assist` — это retrieval-augmented-generation слой, встроенный в саму CLI. Он существует, чтобы операторы и AI-агенты могли задавать вопросы про платформу — «как онбордить новый сервис?», «что означает эта роль Vault?», «какой runbook покрывает этот сценарий отказа?» — и получать указатели в курируемый корпус вместо ручного grep по markdown-файлам.

Этот документ описывает архитектуру с достаточной детализацией, чтобы донести design-выборы, без раскрытия деталей реализации, которые бы привязали описание к конкретному кластеру.

---

## Цели

1. **Retrieval должен быть измеримым.** «Вроде работает» — не критерий релиза; качество запирается в CI.
2. **Корпус — это контракт, а не папка.** Каждый документ в корпусе имеет структурированные метаданные, на которые могут опираться downstream-потребители.
3. **Lexical — default, semantic — опция.** Для bootstrap-контекста платформы BM25 дёшев, детерминирован и audit-friendly. Hybrid retrieval поддерживается, но считается экспериментальным.
4. **Один и тот же интерфейс для людей и агентов.** Никакого отдельного API, никакого отдельного сервиса — `assist` это подкоманда той же CLI, которой пользуются все остальные.
5. **Retrieval-слой никогда ничего не выполняет.** Возвращаемые chunks — это evidence pointers. Решение запустить команду остаётся за оператором или вызывающей системой.

---

## Non-goals

- Это не чат-бот. Встроенного вызова LLM нет. Потребители `assist` (оператор, шаг CI, внешний агент) сами решают как использовать retrieved chunks.
- Это не векторная БД. Система оркестрирует корпус и даёт интерфейс поиска; Qdrant, когда используется, — опциональный backend.
- Это не retrieval-слой для продуктового поиска конечных пользователей. Для такого use case есть отдельный платформенный сервис («Retrieval API») с другим стеком, другими SLO и другой стратегией fusion; он за рамками этого документа.

---

## Корпус как контракт

Корпус — это директория runbook платформы: структурированные markdown-файлы, по одному на runbook, каждый открывается YAML-frontmatter блоком.

Frontmatter **обязательный** и валидируется JSON Schema (`runbook-frontmatter.schema.json`) до ingestion. Обязательные поля:

| Поле | Назначение |
|---|---|
| `id` | Стабильный идентификатор; lookup через `assist runbook <id>` использует его |
| `title` | Человекочитаемое название |
| `domain` | Top-level область (network, security, data, observability, …) |
| `component` | Конкретный компонент внутри домена |
| `lifecycle` | Одно из: `draft`, `active`, `deprecated`, `archived` |
| `status` | Операционная классификация (`stable`, `experimental`, …) |
| `class` | Класс документа (`runbook`, `playbook`, `reference`, …) |
| `safety` | Safety-классификация, влияет на default-видимость в agent retrieval |
| `retrieval_priority` | Численный приоритет для tie-breaking в порядке результатов |
| `owners` | Maintainers (опционально, но рекомендуется) |
| `related` | Перекрёстные ссылки на другие runbook |
| `commands` | Исполняемые сниппеты, которые могут распознавать вспомогательные инструменты |

Документ без валидного frontmatter в индекс не попадает. Это enforced и в `assist validate-corpus`, и в CI; регрессии корпуса поэтому всплывают на merge request, а не на операциях.

`contentHash` chunk-содержимого записывается, чтобы upsert в любой backend (Qdrant) был идемпотентным: повторный ingestion неизменённого корпуса — это no-op.

## Подкоманды

| Подкоманда | Назначение |
|---|---|
| `assist search --q "<topic>"` | Lexical BM25 поиск по корпусу; default-вход для оператора |
| `assist runbook <id>` | Получить полный текст runbook по его id из frontmatter |
| `assist explain` | Сгенерировать объяснение целевого объекта (например, манифеста) с цитатами из корпуса |
| `assist diagnose [--profile <name>]` | Запустить структурированную диагностику с приложенным контекстом из корпуса |
| `assist eval` | Запустить harness eval по golden-вопросам, выдать метрики |
| `assist validate-corpus` | Валидировать frontmatter и chunking по каждому документу |
| `assist export-qdrant` | Произвести дамп корпуса в форму, пригодную для Qdrant ingestion |
| `assist prepare-qdrant-upsert` | Разбить экспорт на upsert-payload нужного размера |
| `assist upstream refresh` | Обновить chunks из upstream-источника документации (например, docs внешнего проекта) в тот же chunk-формат |

## Слои retrieval

### Default: lexical BM25

В bootstrap-контексте — отвечая «где смотреть?» для оператора или агента — предсказуемость важнее максимального recall. Поэтому BM25 — это default:

- in-process токенизатор + inverted index, построенный из chunked-корпуса
- IDF scoring, без зависимости от моделей, без внешнего сервиса
- latency хорошо укладывается в budget agent-loop
- результаты воспроизводимы между запусками и легко аудируются

### Optional: hybrid через Qdrant

Для корпусов достаточно большого объёма, чтобы выигрывать от semantic recall (парафразы, иноязычный ingestion, большие upstream document-set), `assist` может маршрутизировать запросы через hybrid backend:

- dense-векторы через TEI (text-embeddings-inference) sidecar, модель конфигурируется (типично multilingual-embedding такой как `BAAI/bge-m3`)
- sparse-векторы генерятся вместе с dense
- fusion через Reciprocal Rank Fusion (или DBSF), выбирается на этапе конфигурации
- опциональный cross-encoder rerank на top-N кандидатах

Этот путь **read-plane only** и явно помечен в операторской документации платформы как экспериментальный: hybrid-retrieval используется для поиска источников, не для управления выполнением команд.

### Upstream-корпус

Часть вопросов требует знаний, которых нет внутри самой платформы — например, как ведёт себя API конкретного внешнего инструмента. `assist upstream refresh` подтягивает chunked-контент из внешних документационных источников в параллельный индекс под `.platformctl-state/assist/upstream/`, scoped per-product и per-version. Запросы могут опционально включать upstream-источники через флаги.

## Eval как CI-gate

Это, во многом, **самая важная** часть дизайна. Retrieval хрупок: один испорченный frontmatter или искажённый chunk может молча уронить recall. Harness существует, чтобы ловить это на merge request.

### Механизм

- Файл `testdata/queries.yaml` перечисляет курируемые golden-запросы: текст вопроса и ожидаемая цель (path + section).
- `assist eval` прогоняет каждый запрос против сконфигурированного retrieval-пути и считает:
  - **Recall@3** — доля запросов, где ожидаемая цель в топ-3 результатов
  - **MRR@5** — средний reciprocal rank в топ-5
  - **p95 latency** по всему набору запросов
- Команда выходит с non-zero, когда результаты опускаются ниже порогов, переданных как флаги (`--min-recall-at-3`, `--max-p95-latency-ms`).
- CI вызывает `assist eval` на каждом MR, который трогает корпус, схемы или код retrieval.

### Output

`assist eval --output minimal-json` выдаёт per-query rank и pass/fail-флаги, чтобы CI-логи были diff-friendly, а человек-ревьюер мог увидеть какие запросы регрессировали.

### Почему это важно

Без eval-gate RAG превращается в «выглядит правдоподобно» — софт. С ним retrieval держится на тех же инженерных стандартах, что и любая другая CI-gated подсистема: регрессия, которая ушла в прод, — это регрессия, которая ушла из теста, и фикс летит через тот же merge-request пайплайн, что и баг в коде.

## Token economy

Каждая `assist`-подкоманда поддерживает `--output text | json | minimal-json`. Режим `minimal-json` был добавлен явно для LLM-агентов:

- убирает API-envelope (`apiVersion`, `kind`, `metadata.*`), который downstream-потребителям не нужен
- дропает nullable per-result поля, когда они пустые
- оставляет только поля, которыми агенты реально пользуются: matched text, path, score, frontmatter essentials

В измеренных agentic-workflows переключение с `json` на `minimal-json` срезало per-call payload на 60–78%. Для workflow, который вызывает `assist search` и `assist runbook` десятки раз в сессии, это конвертируется в осмысленную экономию токенов.

## Границы с остальным RAG-портфолио

Эта подсистема (`L1` в широкой framing-модели) — один из трёх retrieval-контекстов, которые платформенная команда трактует как разные:

| Слой | Аудитория | Retrieval | Статус |
|---|---|---|---|
| `assist` (этот документ) | Операторы, AI-агенты | Lexical default; experimental hybrid | Production, CI-gated |
| Retrieval API | Продуктовые приложения | Hybrid + опциональный rerank | Foundation; ждёт продуктовых консьюмеров |
| AI-assisted observability | Третья линия поддержки | Vector store знаний; SQL-агрегированные факты | Параллельный domain |

Их смешивание — частая ошибка: у каждого свои SLO, корпуса и операционные ограничения. `assist` намеренно — самый простой из трёх.
